# Miratuku Research Dashboard UX総合レビュー

**実施日:** 2026年4月2日
**対象:** https://yuyanishimura0312.github.io/research-dashboard/
**レビュー体制:** 5人のUX専門エージェントによる並列レビュー

---

## エグゼクティブサマリー

Miratuku Research Dashboardは、リサーチデータの蓄積・分析・探索のための機能が非常に充実したツールである。BM25+ベクトル検索によるセマンティック検索、Louvainコミュニティ検出によるクラスタリング、コメントからのリサーチ方針導出など、単なるリスト管理を超えた知的探索ツールとして高い完成度を持つ。マガジンビューのNYTスタイルレイアウトとリーダーオーバーレイは特に洗練されている。

一方で、機能の増加に伴いタブ構造が7つに拡大し、ナビゲーションの散漫さが目立つ。ダークモードの不完全な実装、モバイル対応の基本的な欠落（ハンバーガーメニュー不在、タッチターゲット不足）、アクセシビリティの根本的な課題（ARIA属性の欠如、キーボード操作不能）が、改善すべき主要領域として浮上した。

## 領域別詳細

### 1. ビジュアルデザイン & ブランディング (3.8/5)

アースートーンのカラーパレット（#7A4033、#FFFAF6、#3A2A20 等）は、知的かつ落ち着いた印象を与え、ミラツクのブランドアイデンティティに非常に適合している。CSS変数による色管理の徹底、clamp()を活用したフルードタイポグラフィ（マガジンビュー）、ドロップキャップによる紙面的品格など、デザインシステムとしての成熟度は高い。

| 項目 | スコア |
|------|--------|
| カラーシステムの一貫性 | 4.0 |
| タイポグラフィ階層と可読性 | 4.5 |
| ビジュアルリズム・スペーシング | 3.5 |
| カードデザインと重量バランス | 4.0 |
| ダークモード実装 | 3.0 |
| ブランドアイデンティティ一貫性 | 4.0 |
| カラーコントラスト・可読性 | 3.5 |

**主な課題:**
- 共有ボタンのブルー系色（#2563EB等）がアースートーンパレットから逸脱
- ダークモードでハードコードされた白背景（rgba(255,255,255,*)）が複数箇所で視覚的破綻
- ログイン画面にCSS変数未導入、ブランドマーク（Mira Tuku + Research二色構成）が不在
- `--text-light: #A08A78` on `#FFFAF6` がコントラスト比約3.2:1でWCAG AA基準（4.5:1）を下回る
- ブランド名表記の揺れ（Miratuku vs Mira Tuku）

### 2. 情報設計 & ナビゲーション (3.3/5)

| 項目 | スコア |
|------|--------|
| タブ構造とナビゲーション | 3.0 |
| 各タブ内の情報階層 | 3.5 |
| 検索・フィルタリングUX | 4.0 |
| コンテンツ発見・探索 | 3.5 |
| DashboardとMagazineの関係性 | 2.5 |
| 管理画面の構成 | 3.5 |
| メンタルモデルとの整合性 | 3.0 |

**主な課題:**

**7タブは多すぎる。** 機能的重複と境界の曖昧さが問題である。

- 「一覧」と「時系列」は同じデータの表示形式違い。一覧タブ内のビュー切り替えにタイムラインを統合可能
- 「ライブラリ」と「構造」は両方ともカテゴリグルーピングで、違いがユーザーに即座にわからない
- 「ホーム」と「分析」は両方とも全体俯瞰が目的で、どちらを見ればよいか迷う

**推奨: 5タブへの再構成**

1. **ホーム** (現ホーム + 分析を統合) -- 俯瞰・インサイト・次のアクション
2. **一覧** (現一覧 + 時系列を統合) -- カード/リスト/タイムラインをビュー切り替え
3. **ライブラリ** (現ライブラリ + 構造を統合) -- トピックマップと階層構造を一画面に
4. **コメント** -- そのまま維持
5. **グラフ** -- ナレッジグラフ可視化（上級者向けオプション）

**その他の課題:**
- 検索が一覧タブにしか存在しない。グローバル検索バーをマストヘッドに配置すべき
- DashboardとMagazineの位置づけが不明確。コメントシステムが二重実装（localStorage vs Firestore）
- タブ名と実際の機能にズレ（「構造」→「分類」が適切）

### 3. インタラクションデザイン & ユーザビリティ (3.1/5)

| 項目 | スコア |
|------|--------|
| ログイン/サインアップフロー | 4.0 |
| カードインタラクション | 4.0 |
| 検索体験 | 4.0 |
| ビュー切り替え | 3.0 |
| 記事リーダーオーバーレイ | 3.0 |
| モーダル/ダイアログ | 3.0 |
| フォームインタラクション | 3.0 |
| フィードバックメカニズム | 2.0 |
| キーボードナビゲーション | 2.0 |
| 状態管理/URL処理 | 3.0 |

**主な課題:**

- **フィードバック機構の欠如が最大の問題。** Firestoreへのデータ保存・削除中に何のフィードバックもない。エラーハンドリングがconsole.errorのみ。トーストメッセージシステムが不在でalert()に依存
- **モーダルの基本機能不足。** bodyのスクロールロックなし、フォーカストラップなし、開閉アニメーションなし
- **URLベースの状態管理が不完全。** history.replaceStateではなくpushStateを使うべき。モーダル状態やフィルタ状態がURLに反映されない
- **キーボードショートカットが2つのみ（Escape、n）で、存在の案内もない**
- **関連リサーチクリック時のモーダル点滅**（閉じて100ms後に再オープン）

### 4. モバイル & レスポンシブデザイン (3.3/5)

| 項目 | スコア |
|------|--------|
| ログインページのモバイルレイアウト | 4.0 |
| タブナビゲーション（7タブ）のオーバーフロー | 3.0 |
| カードグリッドのレスポンシブ | 4.0 |
| タッチターゲットサイズ | 2.0 |
| タイポグラフィスケーリング | 4.0 |
| ツールバー/フィルターコントロール | 3.0 |
| リーダーオーバーレイ | 3.5 |
| グラフ/ビジュアライゼーション | 3.0 |
| 水平スクロール問題 | 3.5 |
| モバイルファースト設計方針 | 2.5 |

**主な課題:**

- **ハンバーガーメニューが存在しない。** マストヘッドの7つのナビボタン（Magazine、テーマ切替、Export、Import、新規追加、管理、ログアウト）がモバイルで確実にオーバーフローする。最も基本的なモバイル対応パターンが欠落
- **タッチターゲットが全般的に小さすぎる。** タブボタン（30-36px）、カードアクション（26px）、ブックマーク（28px）、タグ（24px）など、推奨44pxを大幅に下回る要素が多数
- **iPhoneノッチ/Dynamic Island未対応。** `env(safe-area-inset-*)` が全く考慮されていない
- **ツールバーのモバイル表示が冗長。** 7要素が縦積みでファーストビューの大部分を占有
- タブの横スクロールが可能であることの視覚的ヒント（フェードグラデーション等）がない

### 5. アクセシビリティ & パフォーマンス (2.3/5)

| 項目 | スコア |
|------|--------|
| セマンティックHTML | 2.0 |
| ARIA属性とロール | 1.0 |
| カラーコントラスト | 3.0 |
| フォーカス管理/キーボード | 2.0 |
| スクリーンリーダー対応 | 2.0 |
| フォームラベル/エラー | 3.0 |
| 画像/アイコン代替 | 2.0 |
| ローディング状態通知 | 1.0 |
| パフォーマンス | 2.0 |
| prefers-reduced-motion | 4.0 |
| 言語属性 | 4.0 |
| スキップナビゲーション | 1.0 |

**主な課題:**

- **ARIA属性がほぼ完全に欠如。** 7タブにrole="tablist"/role="tab"なし。3つのモーダルにrole="dialog"なし。エラーメッセージにrole="alert"なし
- **スキップナビゲーションリンクが全ページに不在**
- **クリック可能なdiv要素にtabindex="0"やrole属性がなく、キーボードでフォーカス・操作不能**
- **app.htmlが6400行の巨大ファイル。** 約1400行のインラインCSS、約4800行のインラインJSでブラウザキャッシュが効かない
- **Firebaseスクリプトにdefer/async属性がなく、レンダリングをブロック**
- **ステータスバッジのコントラスト不足。** `--status-reading: #7A8B6E` 上の白テキストが約3.2:1

## 次のステップ

このレビューに基づき、以下の順序での改善を推奨する。

**フェーズ1（即座に対応可能な改善）:**
P0の4項目 + P1のスキップナビゲーション、ARIA属性追加。これらはユーザーの基本的な操作体験とアクセシビリティを確保する最低限の改善であり、既存の機能を壊すリスクも低い。

**フェーズ2（構造的改善）:**
タブの再構成（7→5）、グローバル検索、URLベースの状態管理。これらは情報設計の根本的な改善であり、ユーザーの日常的な利用体験を大幅に向上させる。ただし、既存の機能構造に手を入れるため、十分なテストが必要。

**フェーズ3（品質・保守性の向上）:**
ダークモード完成度向上、カラーシステム統一、パフォーマンス改善（CSS/JS外部化）。長期的なメンテナンス性とユーザー体験の完成度を高める。
